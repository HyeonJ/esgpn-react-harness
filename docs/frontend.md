# docs/frontend.md — 프론트엔드 공통 규칙

이 문서는 프론트엔드 작업 전반(React, Vue, 정적 HTML 등 무관)에 적용되는 공통 규칙을 정의한다.
프레임워크별 세부 규칙은 별도 문서를 참고한다 (예: `docs/react.md`).

작업 시작 전 반드시 `CLAUDE.md`의 5단계 워크플로우를 따른다.

---

## 0. 작업 시작 전 체크리스트

UI 작업을 시작하기 전에 항상 확인한다:

- [ ] 디자인 시안/스크린샷이 있는가? 없다면 어떤 기준으로 만들 것인가?
- [ ] 디자인 토큰(색상, 간격, 폰트)이 정의되어 있는가?
- [ ] 반응형 기준(모바일/태블릿/데스크탑)이 정해져 있는가?
- [ ] 다국어(i18n)가 필요한가?
- [ ] 접근성(a11y) 요구사항이 있는가?
- [ ] 브라우저 지원 범위는 어디까지인가?

리서치 단계에서 위 항목을 `research.md`에 명시적으로 답한다.

---

## 1. 디자인 토큰

### 원칙
- 색상, 간격, 폰트 크기, 그림자, 라운드 등은 **반드시 토큰으로 정의**하고 그것만 사용한다
- 임의값(magic number)은 금지 — 한 번 쓰는 값이라도 토큰에 추가
- 토큰 이름은 의미 기반(`primary`, `surface`, `danger`)을 우선, 색상명(`blue-500`)은 보조

### 다크모드
- 색상은 시맨틱 토큰으로 (`text-primary`, `bg-surface`) — 라이트/다크 자동 전환
- 다크모드 지원 여부는 프로젝트 시작 시 결정 — 나중에 추가하면 모든 컴포넌트 수정 필요

---

## 2. 반응형 디자인

### 브레이크포인트
일반적인 기준 (Tailwind 기본값과 일치):
- `sm`: 640px (큰 모바일)
- `md`: 768px (태블릿)
- `lg`: 1024px (작은 데스크탑)
- `xl`: 1280px (데스크탑)
- `2xl`: 1536px (큰 데스크탑)

### 원칙
- **모바일 우선(mobile-first)** — 기본 스타일은 모바일, 큰 화면에서 오버라이드
- 고정 픽셀(`width: 1200px`) 금지 — `max-w-*` + `mx-auto` 패턴
- 가로 스크롤 금지 — `overflow-x-hidden`은 증상 가림이지 해결책 아님
- 터치 타겟 최소 44x44px (모바일 접근성)

---

## 3. 접근성 (a11y)

### 필수 항목
- 모든 이미지에 `alt` (장식용은 `alt=""`)
- 모든 폼 입력에 `<label>` 또는 `aria-label`
- 색상만으로 정보 전달 금지 (색맹 대응) — 아이콘/텍스트 병행
- 키보드만으로 모든 기능 사용 가능해야 함 (Tab 순서 자연스럽게)
- 포커스 표시(`focus-visible`)를 제거하지 않기 — 디자인이 마음에 안 들면 커스텀

### 시맨틱 HTML
- `<div onclick>` 대신 `<button>`
- `<div>` 남발 대신 `<header>`, `<main>`, `<nav>`, `<section>`, `<article>`, `<footer>`
- 헤딩 레벨(`h1` → `h2` → `h3`) 건너뛰지 않기
- 리스트는 `<ul>`/`<ol>` 사용

### ARIA
- 시맨틱 HTML로 해결되면 ARIA 쓰지 않기 — "ARIA 안 쓰는 것이 잘못된 ARIA보다 낫다"
- Radix UI / Headless UI 같은 라이브러리가 처리하는 ARIA는 신뢰

---

## 4. 폼 처리

### UX 원칙
- 필수 필드는 시각적으로 명시 (`*` 또는 텍스트)
- 검증 에러는 **필드 옆**에 표시 — 토스트만 띄우면 어디가 틀렸는지 모름
- 제출 중에는 버튼 비활성화 + 로딩 표시 — 중복 제출 방지
- 긴 폼은 섹션 분리 또는 단계별(stepper)로 분할
- 입력 중 검증은 `onBlur` 또는 디바운스 — 매 키 입력마다 에러 표시 금지

### 검증
- 클라이언트 검증은 UX용이고, **서버 검증이 진실의 원천** — 클라이언트 검증만 믿지 않기
- 검증 규칙은 가급적 백엔드와 공유 (Zod 스키마 등)
- 에러 메시지는 사용자 행동을 안내 ("이메일 형식이 올바르지 않습니다" → "예: name@example.com")

---

## 5. 로딩 / 에러 / 빈 상태

모든 비동기 화면은 **4가지 상태**를 명시적으로 처리한다:

1. **로딩(Loading)** — 스켈레톤 또는 스피너
2. **성공 + 데이터 있음** — 정상 화면
3. **성공 + 데이터 없음(Empty)** — 빈 상태 메시지 + 다음 행동 제안
4. **에러(Error)** — 에러 메시지 + 재시도 버튼

### 흔한 실수
- 로딩 상태를 빈 화면으로 두기 → 사용자가 멈춘 줄 안다
- 빈 상태를 처리하지 않아 그냥 빈 컨테이너만 보임
- 에러 발생 시 무한 스피너
- 에러 메시지에 기술 용어 노출 (`500 Internal Server Error`)

---

## 6. 성능

### 측정 우선
**측정 후 최적화**한다. 미리 최적화하지 않는다. Lighthouse, Chrome DevTools Performance 탭 활용.

### 일반 원칙
- 이미지: 적절한 포맷(WebP/AVIF), 적절한 크기, `loading="lazy"`
- 폰트: `font-display: swap`, 서브셋팅, 가급적 자체 호스팅
- 코드 스플리팅: 라우트 단위, 무거운 모달/차트는 `lazy` 로딩
- 큰 리스트는 가상화 검토
- 번들 크기 모니터링 — 새 패키지 추가 시 영향 확인

### 금지사항
- 측정 없이 무지성 메모이제이션
- 모든 이미지를 base64 인라인
- 사용하지 않는 라이브러리 import (트리 셰이킹 신뢰하지 말고 import 자체를 줄이기)

---

## 7. 보안

### 필수
- **XSS**: 사용자 입력을 `innerHTML` / `dangerouslySetInnerHTML`에 직접 넣지 않기. 필요하면 sanitize (DOMPurify)
- **민감 정보**: API 키, 토큰, 비밀번호를 클라이언트 코드/로컬스토리지에 절대 저장 금지
- **인증 토큰**: HttpOnly 쿠키 권장, localStorage는 XSS에 취약
- **외부 링크**: `target="_blank"`에는 `rel="noopener noreferrer"`
- **CSP**: 가능하면 Content Security Policy 헤더 적용

### 환경 변수
- `VITE_*` / `REACT_APP_*` 등 빌드 시 노출되는 변수에 비밀 정보 넣지 않기 — 빌드 결과물에 그대로 박힘
- 비밀 키가 필요하면 백엔드 프록시 경유

---

## 8. 국제화 (i18n)

프로젝트 시작 시 i18n이 필요한지 결정한다 — 나중에 추가하면 모든 텍스트를 추출해야 한다.

### 원칙
- 하드코딩된 사용자 대면 텍스트 금지 — 모두 키로 관리
- 날짜/숫자/통화는 `Intl` API 사용 (`Intl.DateTimeFormat`, `Intl.NumberFormat`)
- 텍스트 길이가 언어별로 다름 — 고정폭 레이아웃 피하기 (특히 독일어, 일본어)
- 복수형 처리는 i18n 라이브러리에 위임 (영어의 `1 item / 2 items` 등)

---

## 9. 자산 관리

### 이미지
- 정적 이미지: `public/` 또는 `assets/` — 빌드 도구 규약에 따름
- 컴포넌트와 함께 쓰이는 이미지는 컴포넌트 폴더 안에
- SVG는 가급적 컴포넌트로 import (색상/크기 동적 변경 가능)
- 큰 이미지는 CDN 또는 이미지 최적화 서비스

### 아이콘
- 아이콘 라이브러리(lucide, heroicons 등) 한 가지로 통일
- 아이콘마다 다른 라이브러리 섞어 쓰지 않기 — 디자인 일관성 깨짐 + 번들 크기 증가

---

## 10. 라우팅

### 원칙
- URL은 사용자에게 보이는 인터페이스 — 의미 있고 공유 가능해야 함
- 검색/필터 조건은 URL 쿼리스트링에 반영 (새로고침/공유 시 상태 유지)
- 비공개 라우트는 인증 가드로 보호
- 404 페이지 반드시 정의

### 금지사항
- 모달을 라우트로 만들 때 뒤로가기 동작 깨뜨리기
- 인증 실패 시 무한 리다이렉트 루프

---

## 11. 자주 발생하는 안티패턴 (피할 것)

1. **디자인 시안 없이 "느낌대로" 만들기** — 토큰/간격이 제멋대로가 됨
2. **반응형을 나중에 추가** — 데스크탑 전용으로 만들고 모바일 대응이 지옥이 됨
3. **로딩/빈/에러 상태를 마지막에 추가** — 처음부터 4가지 상태로 설계
4. **클라이언트 검증만 신뢰** — 서버 검증이 진실의 원천
5. **접근성을 마지막에 추가** — 시맨틱 HTML과 ARIA는 처음부터
6. **로컬스토리지에 토큰 저장** — XSS에 취약
7. **이미지 크기 최적화 안 함** — Lighthouse 점수 폭락
8. **하드코딩된 텍스트** — i18n 도입 시 전수 작업 필요
9. **z-index 인플레이션** — `z-index: 9999`가 등장하면 설계가 잘못됨

---

## 12. 작업 시작 시 Claude에게

이 문서를 읽었다면 프론트엔드 작업 시작 전에 다음을 수행한다:

1. 사용 중인 프레임워크의 세부 문서(`docs/react.md` 등)도 함께 읽는다
2. `research.md`에 디자인 토큰, 기존 컴포넌트, 라우팅 구조를 정리
3. 위 체크리스트(섹션 0)에 명시적으로 답변
4. 새 패키지/라이브러리는 사용자 승인 후 도입 (`CLAUDE.md` 5.2)
5. 4가지 상태(로딩/성공/빈/에러)와 반응형, 접근성을 `plan.md`에 명시

`CLAUDE.md`의 4단계 방어 체계는 이 문서와 함께 작동한다.
