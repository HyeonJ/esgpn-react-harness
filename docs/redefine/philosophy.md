# 하네스 철학 — 북극성 문서

> 이 문서는 하네스의 **최종 목표(north star)** 를 정의한다.
> 규칙·게이트·에이전트가 흔들릴 때 돌아와 기준을 재확인하는 문서.
> 바꾸려면 먼저 여기부터 바꾼다.

---

## 1. 우리는 무엇을 만들고 있는가

**Figma → React 변환 하네스.**
AI(Claude)가 섹션 단위로 Figma 디자인을 React 코드로 옮기는 작업을, **일관된 품질 기준 아래 반복 가능하게** 돌리는 작업 프레임.

---

## 2. North Star — "편집 가능한 고충실도"

하네스가 최종적으로 만들어야 하는 결과물은:

> **디자인 의도를 담으면서, 납품 이후에도 수정·확장할 수 있는 코드.**

이 한 문장이 모든 규칙의 상위 기준이다.

### "편집 가능"이 의미하는 것
- 시맨틱 HTML — `<section>`, `<h2>`, `<nav>` 등으로 의도 표현
- 디자인 토큰 참조 — 마법 숫자 대신 `var(--space-6)`, `bg-brand-500`
- 텍스트는 텍스트로 — 검색·번역·접근성 가능
- 반응형 가능한 구조 — `absolute` 난무 금지
- 리팩터 가능한 계층 — div 깊이 최소, 의미 단위로 묶임

### "고충실도"가 의미하는 것
- 치수 ±4px 이내 디자인 의도 유지
- 색상은 디자인 토큰과 일치
- 에셋은 빠짐없이 존재
- 레이아웃 순서·위계 정확

**"편집 가능"이 선행이고, "고충실도"는 그 위에서 최대한 달성한다.**
둘이 충돌하면 편집 가능이 이긴다.

## 3. 픽셀 1:1은 "목표"가 아니라 "증빙"이다

시각 diff(G1)는 **얼마나 닮았는지 보고하는 계기판**이지, 합격/불합격을 판정하는 게이트가 아니다.

| 역할 | 도구 |
|---|---|
| **합격 기준** (차단 게이트) | 구조 품질 — 시맨틱·토큰·i18n·치수·색상 |
| **증빙 수단** (참고 지표) | 시각 diff, Lighthouse, 에셋 존재 |

### 왜 이렇게 뒤집었나

1. **코드는 납품 시점이 끝이 아니라 시작이다.**
   Figma → 코드는 프로젝트의 1%. 나머지 99%는 데이터 바인딩·디자인 변경·A/B 테스트·접근성 보강. 픽셀을 맞추려다 수정 불가능한 코드를 만들면 팀이 그걸 버리고 새로 쓴다.

2. **픽셀 1:1은 흔히 "유지보수 지옥 패턴"을 동반한다.**
   `absolute` 난무, 마법 숫자 하드코딩, 텍스트를 이미지로 박제 — 이게 G1 최소화의 부산물이다. ESGPN에서 `main-hero` 카드·`contest-benefits` CTA가 그 함정의 실례다.

3. **디자이너도 틀릴 수 있고, 디자인 시스템은 진화한다.**
   `padding: 23px`를 그대로 박으면 나중에 `spacing-6 (24px)`로 바꿀 때 검색·치환이 불가능하다. 토큰으로 옮기면 diff 0.3% 발생하지만 **바꿀 수 있는 구조**가 된다. 성숙한 팀은 후자를 원한다.

4. **업계 트렌드는 "의도 고정"으로 가고 있다.**
   Figma Dev Mode가 `variables` 중심으로 간 이유. 픽셀 고정이 아니라 디자이너가 규칙을 정하고 개발자가 그 규칙을 구현한다. 픽셀 1:1을 최종 목표로 삼으면 이 흐름과 역방향이다.

## 4. 게이트 위계

```
[차단 게이트]  FAIL이면 섹션 미완료
  ├─ G5 시맨틱 HTML       (jsx-a11y error 0)
  ├─ G6 텍스트:이미지 비율 (text-bearing raster 0)
  ├─ G8 i18n 가능성        (literal text 허용 리스트 외 0)
  ├─ G2 치수               (±4px)
  └─ G4 색상 토큰 일치율   (≥ 90%)

[참고 지표]  수치만 보고, 합격 기준 아님
  ├─ G1 시각 diff          (목표 ≤ 15%, 완화 ≤ 20%)
  ├─ G3 에셋 존재          (naturalWidth > 0)
  └─ G7 Lighthouse         (a11y ≥ 95, SEO ≥ 90)
```

- 승인 게이트에서 사용자에게 보여줄 때도 **차단 게이트 먼저, 참고 지표는 뒤.**
- G1이 20% 초과하면 "의도를 못 담은 것일 수 있음" 신호 — **사용자 판단**으로 승인/재작업 결정.
- G1만 좋고 차단 게이트 FAIL인 경우 = **자동 반려.** "픽셀은 비슷한데 구조가 망가진 코드"는 가치가 없다.

## 5. 결정 원칙 (충돌 시 참조)

**의심스러울 땐 이 순서로 판단한다:**

1. **수정 가능한가?** — 토큰·시맨틱·텍스트가 살아있는가
2. **디자인 의도를 담았는가?** — 치수·색상·레이아웃 순서
3. **눈으로 얼마나 닮았는가?** — G1 수치

1이 깨지면 2·3이 좋아도 반려.
2가 깨지면 3이 좋아도 재작업.
3은 **보고만 하고 결정은 사람이 한다.**

## 6. 이 문서의 효력

- 새 규칙·게이트·에이전트 프롬프트를 만들 때 **이 문서와 충돌하지 않는지** 확인.
- 충돌하면 둘 중 하나 — (a) 새 규칙을 폐기하거나, (b) 이 문서를 먼저 갱신하고 근거를 기록.
- 이 문서 없이 규칙을 추가하지 않는다. 규칙은 철학의 파생물이어야 한다.
