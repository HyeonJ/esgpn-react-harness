# docs/backend.md — 백엔드 공통 규칙

이 문서는 백엔드 작업 전반(Spring, FastAPI, Node 등 무관)에 적용되는 공통 규칙을 정의한다.
프레임워크별 세부 규칙은 별도 문서를 참고한다 (예: `docs/spring.md`).

작업 시작 전 반드시 `CLAUDE.md`의 5단계 워크플로우를 따른다.

---

## 0. 작업 시작 전 체크리스트

새 기능을 만들기 전에 항상 확인한다:

- [ ] 비슷한 엔드포인트/서비스가 이미 존재하는가?
- [ ] 새 테이블/컬럼이 필요한가? 기존 스키마를 재사용할 수 없는가?
- [ ] 이 변경이 기존 API 계약을 깨뜨리는가? (Breaking change 여부)
- [ ] 인증/인가가 필요한가? 어떤 권한이 필요한가?
- [ ] 외부 시스템(다른 API, 메시지 큐 등)과의 연동이 있는가?
- [ ] 트랜잭션 경계는 어디인가?
- [ ] 예상 트래픽과 데이터 규모는?

리서치 단계에서 위 항목을 `research.md`에 명시적으로 답한다.

## 1. API 설계

### REST 원칙
- 리소스 중심 URL: `/api/v1/{resource(복수)}/{id}/{sub-resource}`
- 동사는 HTTP 메서드에, 명사는 URL에
- 좋음: `POST /api/v1/orders`
- 나쁨: `POST /api/v1/createOrder`

### 버전 관리
- URL 버전(`/v1/`) 또는 헤더 버전 — 프로젝트에서 한 가지로 통일
- Breaking change는 새 버전에서, 기존 버전은 일정 기간 유지
- 비파괴적 변경(필드 추가)은 같은 버전에서 OK

### 멱등성
- `GET`, `PUT`, `DELETE`는 멱등해야 함 — 여러 번 호출해도 같은 결과
- `POST`는 비멱등 — 결제 등 중요 작업은 멱등 키(Idempotency-Key) 헤더 검토

### 응답 포맷
프로젝트 전체에서 일관된 응답 포맷을 사용한다. 한 번 정해진 포맷은 신중히 변경한다.

```json
{
  "success": true,
  "data": { ... },
  "error": null
}
```

또는 에러 시:
```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "USER_NOT_FOUND",
    "message": "사용자를 찾을 수 없습니다"
  }
}
```

### 페이징
- **커서 기반(input-based)**을 우선 검토 — 대용량 데이터, 실시간 추가에 안정적
- **오프셋 기반**은 관리자 화면 등 단순한 경우에만
- 응답에 `nextCursor`, `hasNext`, `total`(가능한 경우만) 포함
- 정렬 기준이 페이징 안정성을 결정 — 유일한 컬럼(id 등)을 정렬에 포함

## 2. 데이터베이스

### 스키마 설계
- 테이블 이름은 **복수형**, **snake_case** (`users`, `complaint_tickets`)
- 컬럼 이름은 **snake_case**
- PK는 `id` 권장 (BIGINT/BIGSERIAL 또는 UUID)
- FK는 `{참조테이블단수}_id` (`user_id`)
- 모든 테이블에 `created_at`, `updated_at` 권장
- 소프트 삭제가 필요하면 `deleted_at` (NULL = 활성)

### 마이그레이션
- 스키마 변경은 **반드시 마이그레이션 도구**로 (Flyway, Liquibase, Alembic 등)
- 운영 DB에 직접 ALTER 금지
- 마이그레이션은 한 방향(forward) 우선 — 롤백 SQL은 가급적 작성하되 의존하지 않기
- 큰 테이블의 컬럼 추가는 NULL 허용으로 시작 → 데이터 백필 → NOT NULL 변경 (3단계)
- 인덱스 추가는 운영 영향 고려 (PostgreSQL은 `CREATE INDEX CONCURRENTLY`)

### 쿼리 작성
- `SELECT *` 금지 — 항상 필요한 컬럼만
- N+1 쿼리 주의 — 컬렉션 조회 시 JOIN 또는 배치 조회
- 인덱스를 의식 — `WHERE`, `JOIN`, `ORDER BY` 컬럼은 인덱스 후보
- 동적 정렬/필터는 화이트리스트 검증 (SQL 인젝션 방지)

### 트랜잭션
- 트랜잭션은 가능한 한 짧게
- 외부 API 호출(HTTP, 메시지 발행 등)은 트랜잭션 안에 넣지 않기 — 롤백 불가 + 커넥션 점유
- 외부 시스템과의 일관성이 필요하면 Outbox 패턴 검토
- 격리 수준은 기본값(READ COMMITTED 등) 유지, 변경 시 `plan.md`에 이유 명시

## 3. 인증과 인가

### 인증 (Authentication — 누구인가)
- 토큰 기반(JWT) 또는 세션 기반 — 프로젝트에서 한 가지로 통일
- 비밀번호는 해시(bcrypt, argon2) — 평문 저장 절대 금지
- JWT 사용 시: 짧은 access token + refresh token, 서명 검증 필수
- 토큰 만료/갱신 정책을 명확히 — 무한 갱신 금지

### 인가 (Authorization — 무엇을 할 수 있는가)
- 모든 엔드포인트에 명시적인 권한 정의 — "기본 비공개" 원칙
- RBAC(역할 기반) 또는 ABAC(속성 기반) 중 일관된 방식
- 권한 체크는 한 곳(미들웨어/필터/AOP)에서 — Controller에 흩어지지 않게
- "내 데이터만 접근"은 ID 일치 검증 필수 — IDOR 취약점 방지

### 금지사항
- 비밀번호/토큰을 로그에 남기기
- 클라이언트가 보낸 권한 정보를 그대로 신뢰
- 권한 체크를 프론트엔드에만 의존

---

## 4. 입력 검증

### 원칙
- **모든 외부 입력은 검증** — 클라이언트 검증을 신뢰하지 않는다
- 검증은 가장 바깥 레이어(Controller / Handler)에서 1차, Service에서 비즈니스 규칙 검증
- 검증 실패 시 명확한 에러 메시지 (어느 필드가 왜 잘못되었는지)

### 검증 항목
- 타입(문자열/숫자/날짜)
- 길이/범위
- 형식(이메일, URL, 정규식)
- 필수 여부
- 비즈니스 규칙(중복, 존재 여부 등은 Service에서)

---

## 5. 에러 처리

### 원칙
- 비즈니스 예외는 도메인 의미를 가진 커스텀 예외 (`UserNotFoundException`)
- 시스템 예외는 그대로 전파하되 전역 핸들러에서 일관되게 변환
- HTTP 상태 코드를 정확히 사용 (`404`, `409`, `422` 등)
- 에러 응답에 추적 ID(trace ID) 포함 — 로그 매칭

### HTTP 상태 코드 가이드
- `200 OK`: 조회/수정 성공
- `201 Created`: 생성 성공
- `204 No Content`: 삭제/응답 본문 없음
- `400 Bad Request`: 입력 형식 오류
- `401 Unauthorized`: 인증 없음/실패
- `403 Forbidden`: 인증은 됐지만 권한 없음
- `404 Not Found`: 리소스 없음
- `409 Conflict`: 비즈니스 규칙 충돌(중복 등)
- `422 Unprocessable Entity`: 형식은 맞지만 의미 오류
- `429 Too Many Requests`: 레이트 리밋 초과
- `500 Internal Server Error`: 예상치 못한 서버 오류
- `503 Service Unavailable`: 일시적 장애/점검

### 금지사항
- 모든 에러를 `500`으로 반환
- 에러 메시지에 스택 트레이스/내부 경로/SQL 노출
- 에러를 삼키기(`catch (e) {}`)

## 6. 로깅과 관측성

### 로그 레벨
- `ERROR`: 즉시 조치 필요한 장애
- `WARN`: 비정상이지만 처리 가능
- `INFO`: 비즈니스 이벤트 (주문 생성, 로그인 등)
- `DEBUG`: 개발/디버깅용 — 운영에서는 비활성

### 원칙
- 구조화 로깅(JSON) 권장 — 검색/필터링 용이
- 모든 요청에 추적 ID(trace ID, request ID) 부여
- 민감 정보 로깅 절대 금지 (비밀번호, 토큰, 주민번호, 카드번호 등)
- 문자열 연결 대신 파라미터 바인딩 (`log.info("user {}", userId)`)

### 메트릭
- 핵심 지표 노출: 요청 수, 응답 시간, 에러율
- 비즈니스 지표도 측정: 주문 수, 결제 성공률 등
- 외부 의존성(DB, 외부 API) 응답 시간 모니터링

## 7. 외부 시스템 연동

### 원칙
- 외부 호출은 **타임아웃 필수** — 무한 대기 금지
- 재시도 정책: 지수 백오프 + 최대 횟수
- 회로 차단기(Circuit Breaker) 검토 — 장애 전파 방지
- 외부 API 응답을 그대로 노출하지 않기 — 자체 DTO로 변환

### 멱등성
- 외부에 보내는 요청도 멱등 키 사용 검토 (재시도 시 중복 방지)
- 메시지 큐 컨슈머는 멱등하게 설계 — 같은 메시지가 여러 번 도착할 수 있음

### 비밀 정보 관리
- API 키, 비밀번호는 환경 변수 또는 비밀 관리 시스템(Vault, AWS Secrets Manager)
- 코드/설정 파일에 하드코딩 절대 금지
- `.env` 파일은 `.gitignore`

---

## 8. 성능과 확장성

### 측정 우선
**측정 후 최적화**한다. 미리 최적화하지 않는다. APM, 프로파일러, slow query log 활용.

### 일반 원칙
- 캐싱은 무효화 전략을 먼저 설계 — "캐싱 추가는 쉽지만 무효화는 어렵다"
- DB가 가장 흔한 병목 — 인덱스, 쿼리 최적화 우선
- 페이지네이션 없는 목록 조회 금지
- 무거운 작업은 비동기/배치로 분리

### 부하 대응
- 레이트 리밋(rate limiting) 적용
- 동시 요청 제한 (DB 커넥션 풀, 스레드 풀)
- Graceful shutdown — 진행 중인 요청을 마치고 종료

---

## 9. 테스트

### 피라미드
- 단위 테스트 (많이): 비즈니스 로직, 순수 함수
- 통합 테스트 (적당히): DB, 외부 의존성 포함
- E2E 테스트 (적게): 핵심 사용자 시나리오

### 원칙
- 새 기능은 최소 하나의 테스트 케이스
- 버그 수정은 회귀 테스트를 먼저 작성 (실패 확인) → 수정 → 통과 확인
- 테스트는 독립적이어야 함 — 실행 순서에 의존 금지
- 외부 시스템은 mock 또는 testcontainer

## 10. 자주 발생하는 안티패턴 (피할 것)

1. **레이어 우회** — Controller에서 DB 직접 접근, Service에서 HTTP 객체 참조
2. **거대한 Service 클래스** — 한 클래스가 수십 개 메서드, 수천 줄
3. **중복된 엔드포인트** — 같은 일을 하는 API가 여러 개 (리서치 부재)
4. **트랜잭션 안에 외부 호출** — 롤백 불가 + 커넥션 점유
5. **N+1 쿼리** — 컬렉션 조회 시 자식마다 쿼리 발생
6. **에러 삼키기** — `catch (e) {}` 또는 로그만 찍고 정상 처리
7. **하드코딩된 비밀 정보** — API 키, 비밀번호가 코드에 직접
8. **권한 체크 누락** — IDOR 취약점, 다른 사용자 데이터 접근
9. **마이그레이션 없는 스키마 변경** — 환경 간 불일치
10. **무제한 페이징** — 메모리 폭발
11. **로그에 민감 정보** — 사고 시 2차 피해

## 11. 작업 시작 시 Claude에게

이 문서를 읽었다면 백엔드 작업 시작 전에 다음을 수행한다:

1. 사용 중인 프레임워크의 세부 문서(`docs/spring.md` 등)도 함께 읽는다
2. `research.md`에 관련 엔드포인트, 서비스, 테이블, 외부 의존성을 정리
3. 위 체크리스트(섹션 0)에 명시적으로 답변
4. API 계약(요청/응답 스키마), DB 변경, 권한 정책을 `plan.md`에 명시
5. 트랜잭션 경계, 에러 케이스, 로깅 포인트를 빠뜨리지 않는다
6. 새 라이브러리/외부 의존성은 사용자 승인 후 도입 (`CLAUDE.md` 5.2)

`CLAUDE.md`의 4단계 방어 체계는 이 문서와 함께 작동한다.
