## 1. 프로젝트 방향

Figma 디자인 시안을 React 코드로 **1:1 고충실도 구현**하는 프로젝트입니다.
이전 시도가 **구현율 약 60% 지점에서 멈춘 경험**이 있어, 같은 실패를 반복하지 않도록 **"사람이 매번 판단·측정하던 작업을 자동화하는 에이전트 하네스"** 를 먼저 구축한 뒤, 그 위에서 페이지를 찍어내는 방식으로 접근했습니다.

**핵심 판단**: 디자인 충실도는 "감"이 아니라 "측정"으로 관리되어야 한다 → 모든 섹션은 **8개 품질 게이트(시각 diff, 치수, 에셋, 색상, 시맨틱 HTML, 텍스트 비율, Lighthouse, i18n)** 를 통과해야 완료.

> **하네스란?** Claude(AI)가 정해진 순서·규칙·품질 기준에 따라 작업하도록 묶어두는 **작업 프레임**입니다. 역할별 에이전트와 자동 측정 스크립트를 조합해, 사람이 매번 지시·검수하던 일을 구조화된 자동 루프로 돌립니다.

---

## 2. 하네스 구조 (핵심 산출물)

AI가 섹션 단위로 **리서치 → 계획 → 구현 → 측정 → 수정** 루프를 자동 수행하도록 설계된 오케스트레이터입니다.

| 구성 | 역할 |
|---|---|
| **오케스트레이터 스킬 1종** | `figma-to-react` — 전체 워크플로우 조율, 승인 게이트 제어 |
| **워커 에이전트 4종** | 프로젝트 셋업 / 페이지 분해 / 섹션 구현 / 반응형 폴리시 |
| **보조 스킬 4종** | 품질 게이트 / 동적 에셋 처리 / 승인 포맷 / 반응형 폴리시 |
| **자동 가드 스크립트 10+종** | 시각 회귀 측정, 잘림 감지, baked-in PNG 검출, diff 히스토리 추적 등 |

**결과**: 사람은 "각 섹션 계획 승인" 1회만 개입하면, 구현·측정·재시도·커밋이 자동으로 돌아갑니다.

## 4. 남은 과제 — 반응형 대응

현재 Figma 디자인은 **1920 단일 뷰포트**만 제공되어, 1440 / 768 / 375 뷰포트에서 잘림·가로스크롤이 발생합니다.
이를 해소하기 위한 **반응형 폴리시 2차 이터레이션**을 진행 중입니다.

- 신규 `responsive-polish` 스킬 + `responsive-worker` 에이전트 투입
- **`detect-cutoff` 스크립트**: 뷰포트별 잘림을 자동 감지 → auto-fit 루프로 해소
- Header 햄버거 메뉴 등 **Figma에 없는 모바일 UI** 추가
- 게이트 강화: "clip 모드만 써서 숨기는 PASS" 금지, 근본 해결 강제
